## 四、本次对话复盘

### 4.1 问题识别

项目上次大改（2026-05-09 FastAPI 迁移）结束后，只更新了核心代码，没有同步清理旧文件和文档。时隔数天后，整个项目存在"代码是新的，文档是旧的"的双轨现象，对 AI 和人类维护者都会造成混淆。

### 4.2 根因分析

大规模架构迁移往往分多个 session 完成，"清理文档"被推迟到后来，最终成为技术债。install.sh 作为用户唯一安装入口，其完成提示直接影响用户操作，但它与主体代码在不同层次维护，容易被遗忘。

### 4.3 解决过程

- 先全面探索，列出所有不一致点（7 类问题）
- 按依赖关系排序：先删代码 → 修代码 → 改文档 → 同步 install.sh → 补全缺失入口
- 发现 install.sh 冲突后，选择"向前走"而不是"向后退"（不恢复旧代码，而是让 install.sh 适应新架构）
- 最后用 61 项自动化检查验证

### 4.4 经验教训

- **架构迁移后立刻整理文档**，不要留给下次。技术债积累后整理成本更高。
- **install.sh 是安装的唯一真相**，README 的安装部分必须与其保持一致，否则用户会按 README 操作然后卡住。
- **删除代码前检查所有引用**，包括 shell 脚本中对 Python 文件的引用（`exec python xxx.py`），不只是 `import`。

### 4.5 预防措施

- 下次架构迁移结束时，在工作日志里写"待整理：需更新 AGENTS.md / install.sh / README"作为提醒
- 重要的安装脚本变更后，同步检查 README 的对应章节
