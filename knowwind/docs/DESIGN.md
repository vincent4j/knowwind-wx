# KnowWind 产品设计文档

---

## 产品定位

**KnowWind** 是一个决策平台，从多个数据源（微信群、Twitter、小红书等）提取有价值的内容，经策略清洗后汇总展示，并对外暴露 MCP 接口供其他 AI 调用。

**当前形态**：本地运行，单用户，只支持微信数据源。

**将来形态**：SaaS 服务，多用户，支持多种数据源（微信、Twitter、小红书、YouTube 等）。有动手能力的用户本地免费部署；不想折腾的用户付费使用 SaaS 版，同一套代码，两种部署方式。

目标用户：使用任意 AI Agent 的人，不一定懂技术。唯一前提：已安装任意支持工具调用的 AI Agent（Claude Code、Cursor、Windsurf 等）。

## 仓库结构

**当前**：两个仓库

| 仓库 | 说明 |
|------|------|
| `knowwind` | 核心平台：数据库、Web UI、MCP Server、REST API |
| `wx-still` | 微信数据源插件：采集、策略、清洗、推送 |

**将来**：每个数据源一个独立仓库，按需安装

| 仓库 | 说明 |
|------|------|
| `knowwind` | 核心平台（不变） |
| `wx-still` | 微信插件（本地数据源） |
| `knowwind-twitter` | Twitter 插件（云端数据源，KnowWind 直接采集） |
| `knowwind-xiaohongshu` | 小红书插件（云端数据源） |
| `knowwind-youtube` | YouTube 插件（云端数据源） |

## 核心概念

### Insight

从数据源提取的有价值内容单元。

**当前支持的类型**：

| 类型 | 说明 |
|------|------|
| `qa` | 问题与解答 |
| `link` | 文章链接与观点 |
| `topic` | 热点话题讨论 |

**将来**：随数据源增加扩展更多类型（如 `decision`、`announcement`），不影响现有类型。

### 数据源分类

**当前**：只有本地数据源（微信），用户本机运行插件，清洗完推送给 KnowWind。

**将来**：两类数据源并存

| 类型 | 例子 | 运行位置 | 用户操作 |
|------|------|---------|---------|
| 本地数据源 | 微信 | 用户本机 | 安装插件，配置策略，手动触发采集 |
| 云端数据源 | Twitter、小红书、YouTube | KnowWind 服务端 | 只需在界面配置账号/关键词，无需本地安装任何东西 |

微信因 wechat-decrypt 的限制，永远是本地数据源，无法上云。

### 策略

决定「用什么视角提取内容」，本质是一段提示词。

**组成**：
- **策略标签**：预设提示词模板（技术问答群 / 资讯分享群 / 通用群），选定后不可修改，要改需删除重建
- **自定义补充**：用户自然语言追加，随时可完善，合并进提示词
- **历史反馈**：用户点评沉淀的规则，自动追加到提示词，下次采集自动带入

**当前**：策略配置在微信插件自己的界面里，KnowWind 不感知策略细节。

**将来**：云端数据源（Twitter 等）的策略配置在 KnowWind 界面里直接操作，因为采集也在云端进行。本地数据源的策略始终在插件自己的界面里。

### 置信度评分

AI 提取每条 insight 后自我评估，输出 0-100 分 + 简短理由（一句话），展示在卡片上。

评估维度：
- **通用维度**（所有策略）：内容完整度、噪音程度
- **策略专属维度**：由策略标签决定（如技术问答群关注答案是否可操作，资讯分享群关注观点是否有独特性）

## 架构

**当前**：

```
微信插件（本地）
  ↓ 规则粗过滤
  ↓ LLM 精提取
  ↓ REST API Push
┌─────────────────────────┐
│       KnowWind          │
│  ┌─────────────────┐    │
│  │     SQLite      │    │
│  └────────┬────────┘    │
│     ↙           ↘      │
│  Web UI      MCP Server │
└─────────────────────────┘
                  ↓
      外部 AI（OpenClaw 等）
```

**将来（SaaS）**：

```
微信插件（用户本机）    Twitter/小红书（云端采集）
  ↓ REST API Push            ↓
┌─────────────────────────────────┐
│           KnowWind SaaS         │
│  ┌──────────────────────────┐   │
│  │  PostgreSQL（多租户隔离） │   │
│  └────────────┬─────────────┘   │
│         ↙          ↘           │
│      Web UI     MCP Server      │
└─────────────────────────────────┘
                    ↓
        外部 AI / 导出（Markdown、飞书等）
```

### 各层职责

| 层 | 当前技术 | 将来技术 | 职责 |
|----|---------|---------|------|
| 数据源插件 | 各自独立 Python | 各自独立（语言不限） | 采集、策略、清洗、推送 |
| 数据层 | SQLite | PostgreSQL（SaaS） | 存储所有 insights，数据访问层抽象，换库不改业务逻辑 |
| Web UI | FastAPI + Vue 3（CDN） | 同左 | 可视化界面 |
| REST API | FastAPI | 同左，加认证 | 接收插件推送数据 |
| MCP Server | MCP 协议 | 同左 | 对外暴露数据查询接口 |

## 数据模型

### insights

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 唯一标识 |
| user_id | TEXT | 用户 ID（当前本地版固定为 `local`，SaaS 版为真实用户 ID） |
| source | TEXT | 数据源（wechat / twitter / xiaohongshu…） |
| type | TEXT | qa / link / topic |
| title | TEXT | 标题 |
| content | TEXT | 完整内容 |
| score | INTEGER | 置信度评分（0-100） |
| score_reason | TEXT | 评分理由（一句话） |
| occurred_at | INTEGER | 内容发生时间（Unix 时间戳） |
| created_at | INTEGER | 写入时间 |
| exported | INTEGER | 是否已导出（0/1） |
| extra | JSON | 插件自定义扩展字段，KnowWind 不感知内容，原样保存 |

`user_id` 现在是固定值，将来 SaaS 上线直接承载多用户隔离，数据库结构不需要改。

### groups（微信插件维护，不在 KnowWind 核心库）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT | 群唯一标识 |
| name | TEXT | 群名称 |
| strategy_label | TEXT | 策略标签 |
| strategy_extra | TEXT | 用户自定义补充 |
| strategy_feedback | TEXT | 历史反馈（点评沉淀，自动追加） |
| last_fetched_at | INTEGER | 上次采集时刻（增量采集用） |

## 插件规范

每个插件需遵守：

1. **推送接口**：调用 KnowWind REST API 写入数据，目标地址可配置（当前填 `localhost`，将来填 SaaS 域名，插件代码不需要改）
2. **认证**：请求头带 `Authorization` token（当前本地版不校验，SaaS 版校验真实 token）
3. **数据格式**：按 insights 表字段推送，`source` 填插件名，`extra` 可自定义
4. **注册**：插件启动时向 KnowWind 注册自己（插件名 + 配置页面入口），KnowWind 导航自动出现该插件入口（参考 Obsidian 插件设置机制）

插件完全自治：自己管采集时机、策略配置界面、清洗逻辑、模型选择。KnowWind 不感知插件细节。

## 界面设计

### KnowWind 主界面

**当前**（只有微信）：

```
[微信] [全部]                           排序▼  筛选▼

┌─────────────────────────────────────────────────┐
│ 怎么在 Claude 里调用自定义工具？          87分    │
│ 答案完整，有可操作步骤                           │
│ 2026-05-09 14:23  微信·AI大航海  QA   [点评]    │
└─────────────────────────────────────────────────┘

─────────────────────────────────────────────────
[整体点评：今天的结果怎么样？                    ]
```

**将来**（多数据源）：

```
[微信] [Twitter] [小红书] [全部]         排序▼  筛选▼

（同上，卡片格式统一，来源字段区分数据源）
```

界面规则：
- 右上角：采集按钮，固定可见，人工触发增量采集
- 顶部 tab：按数据源切换，「全部」显示所有，新增数据源自动出现新 tab
- 列表：统一卡片格式，AI 自动判断卡片类型（QA / 链接 / 话题）
- 卡片：标题 + 分数 + 简短理由 + 时间 + 来源 + 点评按钮
- 底部：全局对话框（整体点评）
- 排序：按时间 / 按分数
- 筛选：按数据源 / 按类型

### 插件配置界面

参考 Obsidian 插件设置机制：KnowWind 只提供导航容器，插件注册自己的配置页面，KnowWind 不渲染插件配置内容。

**当前**（微信插件配置页）：
- 选群 + 设置策略标签（捆绑完成）
- 自然语言补充策略（可选，随时完善）
- 采集按钮（人工触发，增量采集）

**点评交互**：
- 单条点评：卡片上「点评」按钮，输入自然语言
- 整体点评：底部全局对话框输入
- 点评提交后：AI 完善策略 + 沉淀历史反馈，界面提示「策略已更新，是否立刻重跑？」
- 用户选择「立刻重跑」：用新策略重新提取本批消息，结果实时更新
- 用户选择「下次生效」：本次结果不变，下次采集时带入新策略

**将来**（云端数据源，如 Twitter，配置在 KnowWind 界面里）：
- 配置采集策略（跟踪名人 / 热点 / 关键字）
- 自然语言补充
- 采集由 KnowWind 服务端自动执行

### 首次启动引导

1. 安装完核心平台，浏览器自动打开，界面为空
2. 提示「还没有数据源，点击安装」
3. 展示可用插件列表（当前只有微信，将来更多）
4. 用户点「安装」，KnowWind 自动触发 AI Agent 完成插件安装
5. 安装完成，对应 tab 自动出现在导航里

## 安装与启动

### 安装核心平台

用户复制以下内容发给任意 AI Agent，AI 自动完成所有步骤：

```
帮我安装 KnowWind：clone https://github.com/vincent4j/knowwind，安装依赖，注册开机自启，启动服务，打开浏览器。
```

**将来**：KnowWind 官网提供这段安装 prompt，用户直接复制，不需要去 GitHub。

### 启动

**当前**：开机自启（macOS 用 launchd，Windows 用任务计划程序，Linux 用 systemd），用户无需手动启动。浏览器访问 `localhost:端口` 即可使用。

**将来（SaaS）**：用户直接访问域名，无需本地安装任何东西（微信插件除外）。

### 安装数据源插件

**当前**：用户在 KnowWind 界面里点击安装，KnowWind 自动触发 AI Agent 完成插件安装，全程不需要离开界面，不需要去 GitHub，不需要复制命令。

**将来（云端数据源）**：用户在 KnowWind 界面里直接启用，无需安装任何本地插件。

## LLM 调用逻辑

### 模型选择

**当前**：复用 Claude Code 当前登录账号，用户零配置。

**将来**：
- 支持用户自己配置 API Key（Anthropic、OpenAI 等）
- 不同插件可以使用不同模型（微信插件用 Claude，Twitter 插件用 GPT-4 等）

### 调用位置

**当前和将来**：插件本地调用，KnowWind 不感知。每个插件自己管清洗逻辑和模型选择，完全自治。

### 调用流程

**第一步：规则粗过滤（本地，零成本）**
- 过滤系统消息、纯表情、太短的消息
- 命中问题关键词的进入候选池

**第二步：LLM 精提取（批量，一次调用）**
- 输入：候选消息批量 + 策略提示词（标签模板 + 用户补充 + 历史反馈）
- 输出：结构化 insights（类型、标题、内容、评分、评分理由）

**第三步：推送**
- 插件把 insights 推给 KnowWind REST API

## MCP Server

**当前**：暴露只读接口，供外部 AI 查询数据。

| Tool | 说明 |
|------|------|
| `list_insights` | 查询 insights，支持按数据源、日期、类型、分数过滤 |
| `get_insight` | 获取单条 insight 完整内容 |
| `search_insights` | 全文搜索 |
| `list_sources` | 查询已注册的数据源 |

**将来**：新增导出类 tool，不改现有接口

| Tool | 说明 |
|------|------|
| `export_markdown` | 导出为 Markdown 文件下载到本地 |
| `export_feishu` | 写入飞书文档 |

外部 AI（OpenClaw 等）通过 MCP 接口查询 KnowWind 数据，实现跨工具的决策支持。

## SaaS 扩展路径

现在留好的扩展口，将来做 SaaS 时直接用，不需要改业务逻辑：

| 扩展口 | 当前 | SaaS |
|--------|------|------|
| `user_id` 字段 | 固定值 `local` | 真实用户 ID，多租户数据隔离 |
| 推送地址可配置 | `localhost` | SaaS 域名，插件代码不变 |
| API Authorization header | 不校验 | 校验真实 token |
| 数据访问层抽象 | SQLite | 换 PostgreSQL，业务逻辑不动 |

SaaS 上线时新增：用户注册登录系统、多租户隔离、云端部署、Web 服务器。本地版和 SaaS 版同一套代码。

## 技术约束

- 微信数据源支持 Windows、macOS、Linux（wechat-decrypt 支持三平台），永远是本地数据源
- 本地版不上传任何数据到外部服务器
- Python 生态（FastAPI + SQLite）
- 前端：Vue 3（CDN 引入），不需要 Node.js 构建
